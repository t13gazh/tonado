# Tonado Zwischenstandsanalyse — 2026-03-28

## Zusammenfassung

Erste Hardware-Session auf Pi Zero W. App läuft, Audio funktioniert, aber es gibt Performance-Probleme und Security-Lücken die vor einem produktiven Einsatz behoben werden müssen.

| Bereich | Bewertung | Handlungsbedarf |
|---------|-----------|-----------------|
| Performance | **WARNUNG** | MPD frisst 28% RAM + 25% CPU im Idle |
| Speicherplatz | **WARNUNG** | 91% belegt auf 4GB SD, kein Platz für Medien |
| Dependencies | **OK** | 27 Python-Pakete, kein Bloat |
| Security | **KRITISCH** | Kein Auth auf API, Path Traversal möglich |
| Code-Qualität | **WARNUNG** | Junk-Dateien, Error-Handling lückenhaft |

## 1. Performance (WARNUNG)

### RAM-Verbrauch (427 MB total, Pi Zero W)

| Prozess | RSS | % RAM | Bewertung |
|---------|-----|-------|-----------|
| MPD | 121 MB | 27.7% | **KRITISCH** — viel zu hoch für Idle |
| Tonado (uvicorn) | 41 MB | 9.4% | OK für Python/FastAPI |
| NetworkManager | 18 MB | 4.1% | Standard |
| ModemManager | 11 MB | 2.5% | Unnötig, kann deaktiviert werden |
| System + Rest | ~50 MB | ~12% | Standard |
| **Frei** | **99 MB** | **23%** | **Knapp** |

**MPD 121 MB ist das Hauptproblem.** Ursache vermutlich: `always_on "yes"` beim httpd-Output hält Encoder-Puffer offen, auch wenn Browser-Audio aus ist. Außerdem hat MPD volle Plugin-Suite geladen (FLAC, OGG, MP3, AAC decoder etc.).

### CPU-Verbrauch (Idle, nichts spielt)

| Prozess | CPU % | Bewertung |
|---------|-------|-----------|
| MPD | 24.5% | **KRITISCH** — Idle sollte ~0% sein |
| Tonado | 11.5% | **WARNUNG** — 1s elapsed_loop + idle_loop |

**MPD 25% CPU im Idle** ist ein schwerwiegendes Problem auf einem Single-Core. Vermutliche Ursache: `always_on "yes"` beim httpd-Output erzeugt permanent Silence-Frames.

**Tonado 12% CPU im Idle** kommt vom `_elapsed_loop` (1s Timer) + WebSocket broadcast. Sollte nur bei aktivem Playback laufen.

### Startup-Zeit

- MPD: 10s (inkl. DB-Scan)
- Tonado: ~20s (Python-Import + Service-Init + Podcast-Seed)
- Gesamt bis App erreichbar: ~30s nach Boot

### Empfehlungen Performance

1. **MPD httpd: `always_on "no"`** — sofort, spart ~100 MB RAM + 25% CPU
2. **elapsed_loop nur bei state==PLAYING** — bereits implementiert, aber broadcast trotzdem
3. **ModemManager deaktivieren** — `sudo systemctl disable ModemManager` spart 11 MB
4. **MPD Plugins beschränken** — nur MP3/OGG/FLAC laden

### Bewertung

- **4 GB SD reicht nicht** — Upgrade auf 16+ GB zwingend nötig für Medien
- **pip + setuptools im venv** — 23 MB können mit `pip cache purge` und `pip uninstall pip setuptools` gespart werden

## 4. Security (KRITISCH)

### API-Authentifizierung

| Endpoint-Gruppe | Auth erforderlich? | Bewertung |
|-----------------|-------------------|-----------|
| /api/player/* | **NEIN** | KRITISCH — jeder im WLAN kann steuern |
| /api/library/* | **NEIN** | KRITISCH — Upload/Delete ohne Auth |
| /api/cards/* | **NEIN** | WARNUNG — Karten-Mappings öffentlich |
| /api/streams/* | **NEIN** | WARNUNG — Sender/Podcasts manipulierbar |
| /api/config/* | **NEIN** | KRITISCH — alle Config-Werte les/schreibbar |
| /api/system/* | **NEIN** | **KRITISCH** — Reboot/Shutdown ohne Auth |
| /api/auth/* | Teilweise | PIN-Login vorhanden, aber nicht erzwungen |
| /ws | **NEIN** | WARNUNG — alle State-Updates öffentlich |

**Das Auth-System (PIN + JWT) existiert, wird aber nirgends erzwungen.** Jeder im WLAN kann alle Endpoints aufrufen, inklusive System-Reboot und Datei-Upload.

### Path Traversal

- **Library Upload** (`/api/library/upload/{folder}`) — Prüfung ob `folder` Path Traversal enthält (../../) fehlt möglicherweise
- **play-folder** — Pfad wird direkt an MPD übergeben, MPD limitiert auf music_directory
- **SPA Fallback Middleware** — Gibt index.html für nicht-API 404s zurück, kein Zugriff auf beliebige Dateien

### SQL Injection

- aiosqlite mit parametrisierten Queries — **OK**
- Kein Raw-SQL mit String-Formatting gefunden

### Stream Proxy

- `/api/player/stream` proxied MPD HTTP-Output — **WARNUNG**: Kein Rate-Limiting, kein Auth, könnte als Bandbreiten-Amplifier missbraucht werden

### Empfehlungen Security

1. **Auth-Middleware erzwingen** — Alle /api/* Endpoints außer /api/health und /api/auth/login hinter JWT
2. **Path-Validation** — Alle Pfad-Parameter gegen Traversal prüfen
3. **Rate-Limiting** — Besonders für Upload und Stream-Proxy
4. **WebSocket Auth** — Token bei Connect prüfen
5. **CORS konfigurieren** — Aktuell keine CORS-Policy gesetzt

## 5. Code-Qualität (WARNUNG)

### Junk-Dateien

Dateien wie `0`, `None`, `dict`, `Path`, `e.key` wurden ins Repo committed. Ursache: Python-Ausdrücke die versehentlich als Dateinamen interpretiert wurden (vermutlich MPD-Response-Objekte die nach stdout geschrieben wurden). Bereits bereinigt und in .gitignore.

### Error-Handling

- `except Exception` an vielen Stellen — verschluckt Fehler
- `_sync_state` loggt Errors aber fährt fort — könnte zu inkonsistentem State führen
- RSS-Parsing (`_parse_rss`) nutzt `urllib.request` synchron in async Context — blockiert Event Loop

### Architektur

- Services sind sauber getrennt (Player, Card, Config, Stream, Timer etc.)
- Event Bus Pattern für lose Kopplung — gut
- WebSocket Hub für Real-Time Updates — gut
- Kein zirkulärer Import erkannt
- `main.py` Lifespan-Funktion ist sehr lang (~200 Zeilen) — sollte in Setup-Module aufgeteilt werden

## Zusätzliche Findings (Code-Analyse Agent)

### Security — Zusätzlich KRITISCH

- **Config-API leakt Secrets:** `GET /api/config/` gibt ALLE Config-Werte zurück, inkl. `auth.jwt_secret` und `auth.pin_hash.*`. Jeder im WLAN kann JWT-Secret auslesen und sich eigene Tokens generieren.
- **Path Traversal mit shutil.rmtree:** `DELETE /api/library/folders/../../` könnte beliebige Verzeichnisse löschen. Kein `..`-Check auf GET/DELETE-Endpoints.
- **Upload ohne Size-Limit:** Ein einzelner Upload kann die 4GB SD-Karte füllen.
- **RSS XML Bomb:** `xml.etree.ElementTree` ist anfällig für Billion-Laughs-Attacke bei User-hinzugefügten Feeds.

### Performance — Zusätzlich

- **Gyro 50 Hz Polling:** `asyncio.sleep(0.02)` = 50 I2C-Reads/Sekunde über run_in_executor. 20 Hz reicht für Gesten.
- **Playlist in jedem Broadcast:** `_sync_state()` lädt bei jedem Aufruf die komplette Playlist-Info von MPD. Bei 50 Tracks × 1 Broadcast/Sekunde = unnötige Last.
- **Zwei SQLite-Connections:** ConfigService + geteilte Connection auf gleiche DB. Unnötige Ressourcen.
- **elapsed_loop Race Condition:** Zwei Tasks schreiben auf `self._state.elapsed` ohne Lock.

### Code-Qualität — Zusätzlich

- **Encapsulation-Verletzung:** `playlists.py` greift direkt auf `player._connected` und `player._client` zu.
- **`asyncio.get_event_loop()` deprecated:** Sollte `get_running_loop()` sein (Python 3.12+).
- **.gitignore Bug:** `ui/build/` statt `web/build/` — Frontend-Build könnte versehentlich committed werden.
